# TDD・仕様駆動開発ワークフロー

## ワークフロー

### 0. 仕様確認（spec-first）

1. `docs/specs/` の対象機能の仕様書を確認する。なければ `_template.md` から作成
2. 仕様に基づき `docs/openapi.yaml` と `prisma/schema.prisma` を更新（必要な場合）
3. 仕様が不明確な場合はユーザーに確認する

### 1. RED: 失敗するテストを書く

- 仕様に基づいてテストを1つだけ作成する
- テストを実行し、失敗することを確認する
- テストが失敗しない場合は、テストが正しいか見直す

### 2. GREEN: 最小限の実装

- テストを通すための最小限のコードのみ書く
- テストを実行し、パスすることを確認する
- 余計な機能やリファクタリングはここではしない

### 3. REFACTOR: コード改善

- 全テストがパスする状態を維持しながらコードを整理する
- リファクタリング後にテストを再実行して全パスを確認する

### 4. 次のサイクルへ

- 手順1に戻り、次のテストを追加する
- 完了後: `npm run lint && npm run format:check`
- 整合性チェック: spec・openapi・prisma 間の矛盾がないか確認

## テスト規約

- describe: 対象の関数またはクラス名
- it/test: `"should [期待する振る舞い] when [条件]"`
- AAA パターン: Arrange → Act → Assert

## 禁止事項

- テストを実装に合わせて修正すること（実装を直す）
- 仕様変更が必要な場合はユーザーに確認してから仕様を更新する
- `test.skip()` や `xtest()` を残すこと
- テストファイルでの `any` 型の使用
- 1つのテストで複数の振る舞いを検証すること
- 実装の内部詳細に依存するテスト
