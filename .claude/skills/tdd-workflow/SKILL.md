# TDD・仕様駆動開発ワークフロー

## 基本原則

- すべての実装の前にテストを作成する
- Red → Green → Refactor サイクルを厳密に守る
- 1度に1つのテストのみ追加する
- 仕様（specs）を実装の根拠とする

## ワークフロー

### 0. 仕様確認

- `docs/specs/` の対象機能の仕様書を確認する
- 実装対象の機能仕様を把握してからテストを書く
- 仕様が不明確な場合はユーザーに確認する

### 1. RED: 失敗するテストを書く

- 仕様に基づいてテストを1つだけ作成する
- テストを実行し、失敗することを確認する（RED）
- テストが失敗しない場合は、テストが正しいか見直す

### 2. GREEN: 最小限の実装

- テストを通すための最小限のコードのみ書く
- テストを実行し、パスすることを確認する（GREEN）
- 余計な機能やリファクタリングはここではしない

### 3. REFACTOR: コード改善

- 全テストがパスする状態を維持しながらコードを整理する
- 重複の除去、命名の改善、構造の整理を行う
- リファクタリング後にテストを再実行して全パスを確認する

### 4. 次のサイクルへ

- 手順1に戻り、次のテストを追加する

## テスト規約

### 命名規則

- describe: 対象の関数またはクラス名
- it/test: `"should [期待する振る舞い] when [条件]"`

### AAAパターン

```
// Arrange（準備）
const input = createTestInput()

// Act（実行）
const result = targetFunction(input)

// Assert（検証）
expect(result).toBe(expected)
```

## 禁止事項

- テストを実装に合わせて修正すること（実装を直す）
- 仕様変更が必要な場合はユーザーに確認してから仕様を更新する
- `test.skip()` や `xtest()` を残すこと
- テストファイルでの `any` 型の使用
- 1つのテストで複数の振る舞いを検証すること
- 実装の内部詳細に依存するテスト
